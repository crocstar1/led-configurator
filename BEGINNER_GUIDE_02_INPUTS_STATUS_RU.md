# Чтение Data-входов и вычисление статусов портов

Этот документ подробно разбирает одну критически важную часть прошивки: путь от электрических уровней `Data1..Data8` до состояний `waiting`, `charging`, `error` и `disabled`.

Здесь показан полный актуальный код каждой рассматриваемой функции. После кода объясняется каждая инструкция, изменение переменных, публикация результата и поведение при ошибке.

Исходные файлы этого пути:

- `src/main.cpp` запускает периодическое чтение;
- `src/board_config.h` задает GPIO и число активных портов;
- `src/station_inputs.h/.cpp` читает электрические уровни;
- `src/status_mapper.h/.cpp` превращает входы в статусы портов;
- `src/led_manager.cpp` читает готовые статусы и выбирает цвета матрицы.

---

## 1. Какие данные участвуют в работе

Перед разбором функций нужно понять три структуры и две защищенные рабочие переменные.

### 1.1. `StationInputState`: состояние восьми физических входов

Актуальное объявление из `src/station_inputs.h`:

```cpp
struct StationInputState {
    bool active[BOARD_INPUT_COUNT];
    uint8_t rawLevel[BOARD_INPUT_COUNT];
};
```

`struct` объединяет несколько связанных полей в один тип данных.

Внутри находятся два массива длиной `BOARD_INPUT_COUNT`, то есть 8:

- `rawLevel[0..7]` хранит буквальный электрический уровень `LOW` или `HIGH`;
- `active[0..7]` хранит уже понятный программе признак `true` или `false`.

Например, для активного по низкому уровню Data1:

```text
rawLevel[0] = LOW
active[0]   = true
```

Индекс массива начинается с нуля. Поэтому Data1 имеет индекс 0, Data2 имеет индекс 1, а Data8 имеет индекс 7.

В `src/station_inputs.cpp` существует одна рабочая переменная модуля:

```cpp
static StationInputState inputState = {};
```

Слово `static` на уровне файла означает, что переменная видна только внутри `station_inputs.cpp`. Другой файл не может случайно изменить ее напрямую.

Запись `= {}` заполняет все поля нулями. Для `bool` это `false`, для `uint8_t` это 0.

### 1.2. `PortStatus`: четыре возможных состояния порта

Актуальный тип из `src/status_mapper.h`:

```cpp
enum PortStatus : uint8_t
{
    PORT_STATUS_WAITING = 0,
    PORT_STATUS_CHARGING,
    PORT_STATUS_ERROR,
    PORT_STATUS_DISABLED
};
```

Перечисление `enum` дает понятные имена числовым значениям:

| Имя | Числовое значение | Смысл |
|---|---:|---|
| `PORT_STATUS_WAITING` | 0 | порт ожидает |
| `PORT_STATUS_CHARGING` | 1 | идет зарядка |
| `PORT_STATUS_ERROR` | 2 | есть ошибка |
| `PORT_STATUS_DISABLED` | 3 | порт зарезервирован и не используется |

Тип `PortStatus` нужен, чтобы код не передавал непонятные числа 0, 1, 2 и 3. Он также ограничивает смысл поля только статусами порта.

### 1.3. `PortInputMapping`: какие входы относятся к порту

Актуальная структура из `src/status_mapper.h`:

```cpp
struct PortInputMapping
{
    uint8_t chargingInput;
    uint8_t errorInput;
    uint8_t zoneId;
    bool enabled;
};
```

Она описывает один порт:

- `chargingInput` - индекс сигнала зарядки в `StationInputState.active[]`;
- `errorInput` - индекс сигнала ошибки;
- `zoneId` - номер зоны матрицы этого порта;
- `enabled` - активен порт в данной сборке или зарезервирован.

Например, Port1:

```text
chargingInput = 0  -> Data1
errorInput    = 1  -> Data2
zoneId        = 1  -> зона Port1
enabled       = true
```

Port2:

```text
chargingInput = 2  -> Data3
errorInput    = 3  -> Data4
zoneId        = 2  -> зона Port2
enabled       = true
```

Эта структура отделяет физическое соответствие входов от алгоритма выбора статуса. Функция вычисления может сказать: «возьми вход ошибки этого порта», не записывая вручную Data2, Data4, Data6 и Data8 в нескольких местах.

### 1.4. `StatusMapperState`: полный опубликованный результат

Актуальная структура из `src/status_mapper.h`:

```cpp
struct StatusMapperState
{
    uint8_t activePortCount;
    PortInputMapping ports[MAX_PORT_COUNT];
    PortStatus statuses[MAX_PORT_COUNT];
};
```

Она хранит согласованный результат для всех четырех возможных портов:

- `activePortCount` - сколько портов реально используется;
- `ports[0..3]` - соответствие входов и зон;
- `statuses[0..3]` - текущий статус каждого порта.

В `src/status_mapper.cpp` есть одна рабочая переменная:

```cpp
static StatusMapperState mapperState = {};
```

Она скрыта внутри файла словом `static`. Светодиодный модуль и диагностика не получают к ней прямой доступ. Они запрашивают согласованную копию через `status_mapper_get_snapshot()`.

### 1.5. Почему появились защищенные снимки

Входы читаются в отдельной постоянно выполняемой работе ESP32, которая называется задачей FreeRTOS (`FreeRTOS task`). Светодиоды, веб-сервер и диагностика могут выполняться в другое время и даже на другом ядре процессора.

Если один код записывает восемь элементов массива, а другой одновременно читает их, читатель теоретически может получить смесь старых и новых данных.

Поэтому используется схема:

```text
сначала собрать новое состояние во временной переменной
                        ↓
на очень короткое время закрыть доступ
                        ↓
целиком заменить опубликованное состояние
                        ↓
открыть доступ
```

Согласованная копия состояния называется снимком (`snapshot`). Снимок нужен не ради сохранения в постоянной памяти настроек (NVS), а для безопасного обмена между одновременно работающими частями программы.

Для короткой защиты используются:

```cpp
static portMUX_TYPE inputStateMux = portMUX_INITIALIZER_UNLOCKED;
static portMUX_TYPE mapperStateMux = portMUX_INITIALIZER_UNLOCKED;
```

Тип `portMUX_TYPE` является легкой блокировкой критической секции для ESP32. Он нужен, потому что у ESP32 два ядра и код может обращаться к общей переменной одновременно.

Функция `portENTER_CRITICAL(&inputStateMux)` входит в защищенный участок. Символ `&` здесь означает «передать адрес переменной `inputStateMux`».

Функция `portEXIT_CRITICAL(&inputStateMux)` выходит из защищенного участка.

Внутри такой секции нельзя выполнять долгие операции. В текущем коде там находится только копирование небольшой структуры, поэтому защита короткая.

## 2. Общая карта вызовов

```text
setup()
  |
  +-> station_inputs_setup()
  |     +-> настройка GPIO
  |     +-> station_inputs_update()
  |
  +-> status_mapper_setup()
  |
  +-> updatePortStatusFromInputs()
  |     +-> station_inputs_update()
  |     +-> station_inputs_get_snapshot()
  |     +-> status_mapper_update()
  |
  +-> led_setup()
  |     +-> стартовая проверка ошибки тоже читает входы и статусы
  |
  +-> startInputTask()
        +-> xTaskCreatePinnedToCore(... inputTaskWorker ...)
                |
                +-> каждые примерно 50 мс:
                    updatePortStatusFromInputs()
```

Если создать отдельную задачу не удалось:

```text
loop()
  +-> каждые 50 мс вызывает updatePortStatusFromInputs()
```

## 3. `updatePortStatusFromInputs()`

Файл: `src/main.cpp`, текущие строки 26-31.

### 3.1. Полный актуальный код

```cpp
static void updatePortStatusFromInputs() {
    station_inputs_update();
    StationInputState inputs;
    station_inputs_get_snapshot(inputs);
    status_mapper_update(inputs);
}
```

### 3.2. Кто и когда вызывает функцию

Функция вызывается:

1. Один раз из `setup()` после `station_inputs_setup()` и `status_mapper_setup()`.
2. Постоянно из `inputTaskWorker()` примерно каждые 50 мс.
3. Из `loop()` как запасной путь, если отдельную задачу создать не удалось.

Слово `static` перед функцией означает, что вызвать ее можно только внутри `main.cpp`.

Тип результата `void` означает, что функция не возвращает значение через `return`. Результат публикуется в состояниях модулей входов и статусов.

### 3.3. Какие данные существуют перед входом

До вызова уже существуют:

- массив `BOARD_INPUT_PINS[8]` с GPIO;
- глобальное состояние входов `inputState` внутри `station_inputs.cpp`;
- глобальное состояние портов `mapperState` внутри `status_mapper.cpp`;
- физические уровни на Data1..Data8.

При обычном периодическом вызове `inputState` и `mapperState` содержат результаты предыдущего периода.

### 3.4. Построчное объяснение

#### Строка 1

```cpp
static void updatePortStatusFromInputs() {
```

Объявляется внутренняя для `main.cpp` функция без возвращаемого значения и без параметров.

#### Строка 2

```cpp
station_inputs_update();
```

Вызывается чтение всех восьми физических GPIO. Функция создает новый `StationInputState` и публикует его как `inputState`.

#### Строка 3

```cpp
StationInputState inputs;
```

Создается локальная переменная `inputs`. Она существует только до завершения текущего вызова.

На этой строке ее поля еще не следует читать: явного заполнения `= {}` здесь нет. Следующая строка сразу полностью копирует в нее корректное состояние.

#### Строка 4

```cpp
station_inputs_get_snapshot(inputs);
```

В `inputs` копируется согласованный снимок (`snapshot`) опубликованного `inputState`.

Функция получает переменную по ссылке. Ссылка в C++ записывается символом `&` в объявлении вызываемой функции. Она позволяет заполнить именно локальную переменную `inputs`, а не ее отдельную копию.

#### Строка 5

```cpp
status_mapper_update(inputs);
```

Снимок входов передается вычислителю статусов. Параметр там объявлен как `const StationInputState &inputs`:

- `&` означает передачу по ссылке без копирования всей структуры;
- `const` запрещает `status_mapper_update()` изменять полученный снимок.

#### Строка 6

```cpp
}
```

Локальная переменная `inputs` перестает существовать. Опубликованные `inputState` и `mapperState` продолжают жить.

### 3.5. Какие переменные создаются и меняются

Локально создается только:

```text
inputs : StationInputState
```

Глобально через вызываемые функции меняются:

- `inputState`;
- `mapperState`.

### 3.6. Что функция публикует и кто использует результат

Функция ничего не возвращает напрямую. После нее:

- `station_inputs_get_snapshot()` может отдать новые уровни диагностике;
- `status_mapper_get_snapshot()` может отдать новые статусы светодиодному модулю, диагностике и `/get_config`.

### 3.7. Что произойдет при ошибке

У функции нет собственного кода ошибки.

- Если GPIO электрически подключен неправильно, будет опубликован фактически прочитанный уровень.
- Если сигнал шумит, статусы могут быстро меняться. Программная фильтрация пока не добавлена и должна решаться после проверки железа.
- Если задача не стартовала, эту функцию вызывает запасной код в `loop()`.

## 4. `inputTaskWorker()`

Файл: `src/main.cpp`, текущие строки 33-41.

### 4.1. Полный актуальный код

```cpp
static void inputTaskWorker(void *) {
    TickType_t lastWake = xTaskGetTickCount();
    const TickType_t period = pdMS_TO_TICKS(INPUT_TASK_PERIOD_MS);

    while (true) {
        updatePortStatusFromInputs();
        vTaskDelayUntil(&lastWake, period);
    }
}
```

### 4.2. Кто и когда вызывает функцию

Код не вызывает `inputTaskWorker()` обычной строкой вида `inputTaskWorker()`.

Функция передается как адрес в `xTaskCreatePinnedToCore()`. Планировщик ESP32 начинает выполнять ее как отдельную постоянно работающую задачу FreeRTOS (`FreeRTOS task`).

Она стартует после вызова `startInputTask()` из `setup()`.

### 4.3. Какие данные существуют перед входом

До старта:

- GPIO уже настроены;
- первый снимок входов уже опубликован;
- `status_mapper_setup()` уже выполнился;
- светодиодный модуль уже запущен;
- `INPUT_TASK_PERIOD_MS` равен 50;
- `inputTaskStarted` еще будет обновлен функцией запуска.

### 4.4. Построчное объяснение

#### Заголовок

```cpp
static void inputTaskWorker(void *) {
```

`static` ограничивает видимость файлом `main.cpp`.

Аргумент `void *` является универсальным указателем на необязательные данные задачи. Имя аргумента отсутствует, потому что текущая задача не использует переданный параметр. При создании передается `nullptr`.

#### Начальный момент

```cpp
TickType_t lastWake = xTaskGetTickCount();
```

`TickType_t` хранит число тактов планировщика. Один такт планировщика называется тиком (`tick`). Это внутренняя единица времени FreeRTOS.

Не следует вручную считать, что один тик всегда равен одной миллисекунде. Частота зависит от настройки среды. Поэтому для миллисекунд используется специальное преобразование.

`xTaskGetTickCount()` возвращает текущее число тиков с момента запуска планировщика.

Переменная `lastWake` получает начальную точку расписания задачи.

#### Период

```cpp
const TickType_t period = pdMS_TO_TICKS(INPUT_TASK_PERIOD_MS);
```

`INPUT_TASK_PERIOD_MS` равен 50 миллисекундам.

`pdMS_TO_TICKS()` переводит миллисекунды в количество тиков с учетом конфигурации FreeRTOS. Это делает код независимее от конкретной частоты тиков.

Слово `const` означает, что после создания значение `period` нельзя изменить внутри функции.

#### Бесконечный цикл

```cpp
while (true) {
```

Цикл `while` повторяет блок, пока условие истинно. Условие `true` истинно всегда, поэтому цикл бесконечный.

Для постоянно работающей задачи это нормально. Она не должна завершаться сама.

#### Обновление

```cpp
updatePortStatusFromInputs();
```

Считываются все Data-входы и публикуются новые статусы.

#### Ожидание следующей точки расписания

```cpp
vTaskDelayUntil(&lastWake, period);
```

Функция откладывает задачу до следующего абсолютного момента расписания.

`&lastWake` передает адрес переменной. `vTaskDelayUntil()` обновляет ее на следующий запланированный момент.

Это отличается от обычной задержки «подождать 50 мс после завершения работы».

Если чтение заняло 2 мс:

- обычная задержка 50 мс дала бы период около 52 мс;
- `vTaskDelayUntil()` подождет оставшиеся примерно 48 мс и сохранит период около 50 мс.

### 4.5. Какие переменные создаются и меняются

Локально один раз создаются:

- `lastWake` - изменяется `vTaskDelayUntil()` каждый период;
- `period` - остается постоянным.

Через `updatePortStatusFromInputs()` постоянно меняются `inputState` и `mapperState`.

### 4.6. Что функция возвращает или публикует

Она не возвращает значение и в нормальной работе не завершается. Каждые примерно 50 мс публикует новые входы и статусы через вызванные модули.

### 4.7. Кто использует результат дальше

Новый `mapperState` читает:

- `led_refresh_internal()` для физической матрицы;
- `handleDiagnostics()` для вкладки диагностики;
- `handleGetConfig()` для `activePortCount`;
- проверка обязательных зон;
- формирование метаданных Port1..Port4.

### 4.8. Что произойдет при ошибке или долгом цикле

У функции нет исключений или кода возврата.

Если один вызов обновления займет больше 50 мс, `vTaskDelayUntil()` увидит, что плановый момент уже прошел, и не добавит полную задержку. Задача попытается догнать расписание.

Два экземпляра функции параллельно не запускаются: следующая итерация начинается только после завершения предыдущей.

Если зависнет `digitalRead()` или другой код внутри обновления, эта задача перестанет публиковать новые статусы. В текущей реализации `digitalRead()` является короткой операцией.

## 5. `startInputTask()`

Файл: `src/main.cpp`, текущие строки 43-57.

### 5.1. Полный актуальный код

```cpp
static void startInputTask() {
    const BaseType_t result = xTaskCreatePinnedToCore(
        inputTaskWorker,
        "Input_Task",
        2048,
        nullptr,
        2,
        nullptr,
        1
    );
    inputTaskStarted = result == pdPASS;
    if (!inputTaskStarted) {
        Serial.println("Input task start failed; using loop fallback");
    }
}
```

### 5.2. Кто и когда вызывает функцию

`setup()` вызывает `startInputTask()` один раз после `led_setup()` и до `network_setup()`.

Это важно: чтение входов отделяется до запуска сетевой части.

### 5.3. Какие данные существуют перед входом

- функция `inputTaskWorker` уже скомпилирована и доступна внутри `main.cpp`;
- глобальная переменная `inputTaskStarted` изначально равна `false`;
- FreeRTOS уже доступна внутри среды ESP32;
- входы и статусная модель уже инициализированы.

### 5.4. Построчное объяснение

#### Создание результата

```cpp
const BaseType_t result = xTaskCreatePinnedToCore(
```

Функция `xTaskCreatePinnedToCore()` просит FreeRTOS создать отдельную работу и закрепить ее за выбранным ядром процессора.

`BaseType_t` является целочисленным типом FreeRTOS для результата операции.

`const` запрещает позже менять локальную переменную `result`.

#### Параметр 1

```cpp
inputTaskWorker,
```

Передается функция, которую должна выполнять новая задача.

#### Параметр 2

```cpp
"Input_Task",
```

Это служебное имя задачи. Оно полезно при диагностике и не показывается пользователю.

#### Параметр 3

```cpp
2048,
```

Задаче выделяется собственная область временной памяти для локальных переменных и вызовов функций. Такая область называется стеком (`stack`).

В реализации ESP32 этот параметр задает размер стека задачи. Здесь выделено 2048 байт.

Слишком маленький стек может привести к повреждению памяти и аварийному перезапуску. Слишком большой зря расходует оперативную память.

#### Параметр 4

```cpp
nullptr,
```

Дополнительные данные задаче не передаются. `nullptr` означает «нет указателя на объект».

#### Параметр 5

```cpp
2,
```

Это приоритет задачи (`priority`). Если несколько готовых задач хотят выполняться одновременно, задача с большим числом получает преимущество.

Приоритет чтения входов равен 2. Он выше приоритета светодиодной задачи 1, чтобы короткое чтение Data-входов выполнялось своевременно.

Это не означает, что задача управления светодиодами остановится: задача входов быстро читает GPIO и затем ждет следующего периода.

#### Параметр 6

```cpp
nullptr,
```

Код не просит вернуть дескриптор задачи, то есть отдельный объект для последующего управления ею.

#### Параметр 7

```cpp
1
```

Задача закрепляется за ядром процессора номер 1 (`core 1`). Ядро является одним из двух вычислительных исполнителей ESP32.

Закрепление делает место выполнения предсказуемым. Защищенные снимки все равно нужны, потому что другие задачи могут выполняться параллельно.

#### Проверка запуска

```cpp
inputTaskStarted = result == pdPASS;
```

Выражение `result == pdPASS` дает `true`, если FreeRTOS успешно создала задачу.

Это значение сохраняется в глобальной переменной `inputTaskStarted`.

#### Обработка неудачи

```cpp
if (!inputTaskStarted) {
```

Условие `if` выполняет блок, только если задача не запущена. Символ `!` меняет `false` на `true` для проверки.

```cpp
Serial.println("Input task start failed; using loop fallback");
```

В диагностический порт выводится сообщение о запасном режиме.

### 5.5. Какие переменные создаются и меняются

- локальная `result` создается один раз;
- глобальная `inputTaskStarted` становится `true` или остается `false`.

### 5.6. Что функция возвращает или публикует

Функция имеет тип `void`, поэтому напрямую ничего не возвращает.

Она публикует результат через `inputTaskStarted` и, при успехе, создает работающую задачу.

### 5.7. Кто использует результат дальше

Функция `loop()` проверяет `inputTaskStarted`:

```cpp
if (!inputTaskStarted) {
    // обновлять входы запасным способом
}
```

### 5.8. Что произойдет при ошибке

Создание может не пройти, например из-за нехватки оперативной памяти для стека.

Тогда:

1. `result` не равен `pdPASS`;
2. `inputTaskStarted = false`;
3. печатается сообщение;
4. `loop()` продолжает читать входы примерно каждые 50 мс.

Таким образом, индикация не должна полностью потерять входы только из-за неудачи запуска задачи.

## 7. `station_inputs_update()`

Файл: `src/station_inputs.cpp`, текущие строки 13-27.

### 7.1. Полный актуальный код

```cpp
void station_inputs_update() {
    StationInputState nextState = {};

    for (uint8_t i = 0; i < BOARD_INPUT_COUNT; i++) {
        const uint8_t level = digitalRead(BOARD_INPUT_PINS[i]);
        const bool active = BOARD_INPUT_ACTIVE_LOW ? (level == LOW) : (level == HIGH);

        nextState.rawLevel[i] = level;
        nextState.active[i] = active;
    }

    portENTER_CRITICAL(&inputStateMux);
    inputState = nextState;
    portEXIT_CRITICAL(&inputStateMux);
}
```

### 7.2. Кто и когда вызывает функцию

Она вызывается:

1. из `station_inputs_setup()` при старте;
2. из `updatePortStatusFromInputs()` примерно каждые 50 мс;
3. из `startup_error_active()` во время стартовой цветовой последовательности, чтобы ошибка могла прервать анимацию.

### 7.3. Какие данные существуют перед входом

- восемь GPIO настроены как входы;
- `BOARD_INPUT_ACTIVE_LOW = true`;
- `inputState` хранит предыдущее опубликованное чтение;
- физические линии имеют текущие уровни.

### 7.4. Построчное объяснение

#### Временная структура

```cpp
StationInputState nextState = {};
```

Создается локальная структура и полностью обнуляется.

Она называется `nextState`, потому что это кандидат на следующее опубликованное состояние. Глобальный `inputState` пока не меняется.

#### Цикл по входам

```cpp
for (uint8_t i = 0; i < BOARD_INPUT_COUNT; i++) {
```

Цикл `for` выполняется восемь раз для индексов 0..7.

#### Чтение уровня

```cpp
const uint8_t level = digitalRead(BOARD_INPUT_PINS[i]);
```

`digitalRead()` читает текущий уровень GPIO.

`level` получает `LOW` или `HIGH`. Слово `const` запрещает изменять результат текущего чтения до конца этой итерации.

#### Преобразование по принципу «активен низкий уровень» (active-low)

```cpp
const bool active = BOARD_INPUT_ACTIVE_LOW ? (level == LOW) : (level == HIGH);
```

Принцип «активен низкий уровень» называется active-low. При текущем `BOARD_INPUT_ACTIVE_LOW = true` выражение проверяет `level == LOW`.

Оператор `условие ? значение1 : значение2` является краткой записью `if/else`:

```text
если BOARD_INPUT_ACTIVE_LOW == true:
    active = (level == LOW)
иначе:
    active = (level == HIGH)
```

#### Сохранение сырого уровня

```cpp
nextState.rawLevel[i] = level;
```

В массив временной структуры записывается буквальный уровень.

#### Сохранение логической активности

```cpp
nextState.active[i] = active;
```

В соседний массив записывается результат с учетом принципа «активен низкий уровень» (active-low).

#### Вход в защищенный участок

```cpp
portENTER_CRITICAL(&inputStateMux);
```

Функция получает блокировку типа `portMUX_TYPE`. Символ `&` передает адрес блокировки.

Пока секция закрыта, другая задача не должна одновременно копировать `inputState`.

#### Публикация

```cpp
inputState = nextState;
```

Вся структура копируется одним логическим действием. Публикуются оба массива для всех восьми входов.

#### Выход

```cpp
portEXIT_CRITICAL(&inputStateMux);
```

Защита снимается. Читатели могут получить новый снимок.

### 7.5. Какие переменные создаются и как меняются

Локально:

- `nextState` постепенно заполняется восемью уровнями и признаками;
- `i` идет от 0 до 7;
- для каждой итерации создаются неизменяемые `level` и `active`.

Глобально один раз в конце меняется `inputState`.

### 7.6. Что функция возвращает или публикует

Прямого результата нет. Она публикует новый `inputState`.

### 7.7. Кто использует результат дальше

- `updatePortStatusFromInputs()` сразу получает снимок;
- стартовая проверка ошибки получает снимок;
- `/diagnostics` получает снимок для отображения Data1..Data8.

### 7.8. Что произойдет при ошибке

Кода ошибки нет.

Если уровень изменится в середине цикла чтения, `nextState` может отражать входы, прочитанные в пределах нескольких последовательных микромоментов, а не строго одновременно. Для дискретных сигналов и короткого цикла это приемлемо, но должно быть проверено на реальном оборудовании.

Если вход шумит, каждый новый уровень будет честно опубликован. Фильтрации или задержки подтверждения сейчас нет.

## 9. `status_mapper_setup()`

```cpp
void status_mapper_setup() {
    StatusMapperState nextState = {};
    nextState.activePortCount = DEFAULT_ACTIVE_PORT_COUNT;

    for (uint8_t i = 0; i < MAX_PORT_COUNT; i++) {
        nextState.ports[i].chargingInput = i * 2;
        nextState.ports[i].errorInput = (i * 2) + 1;

    portEXIT_CRITICAL(&mapperStateMux);
}
```

### 9.2. Кто и когда вызывает функцию

`setup()` вызывает ее один раз после первой настройки входов.

`mapperState` существует и сначала заполнен нулями.

### 9.4. Построчное объяснение

#### Временная структура

```cpp
StatusMapperState nextState = {};
```

Создается и обнуляется кандидат на новое состояние.

#### Число активных портов

```cpp
nextState.activePortCount = DEFAULT_ACTIVE_PORT_COUNT;
```

Записывается 2. Это единственный источник числа активных портов для текущего запуска.

```cpp
for (uint8_t i = 0; i < MAX_PORT_COUNT; i++) {
```

Цикл `for` выполняется для `i = 0, 1, 2, 3`.

```cpp
nextState.ports[i].chargingInput = i * 2;
```

Получаются индексы 0, 2, 4, 6, то есть Data1, Data3, Data5, Data7.

```cpp
nextState.ports[i].errorInput = (i * 2) + 1;
```

Получаются индексы 1, 3, 5, 7, то есть Data2, Data4, Data6, Data8.

- `i = 1`: `1 < 2`, значит `true`;

```cpp
portENTER_CRITICAL(&mapperStateMux);
mapperState = nextState;
portEXIT_CRITICAL(&mapperStateMux);
```

На короткое время закрывается доступ, вся структура публикуется и доступ открывается.

Локально:

- `nextState`;

Глобально меняется `mapperState`.

Она не возвращает значение, а публикует:

```text
activePortCount = 2
ports[0..3]      = соответствие (mapping) Data-пар и зон

```

### 9.7. Кто использует результат дальше

`status_mapper_update()` берет это соответствие входов и заменяет начальные статусы ожидания реальными статусами.

### 9.8. Что произойдет при ошибке

Функция не возвращает ошибку. Она предполагает, что:

- `DEFAULT_ACTIVE_PORT_COUNT <= MAX_PORT_COUNT`;

Текущие значения 2, 4 и 8 согласованы. Если разработчик нарушит это соответствие, нужна новая проверка кода и железа.

## 10. `status_mapper_update()`

Файл: `src/status_mapper.cpp`, текущие строки 25-53.

### 10.1. Полный актуальный код

```cpp
void status_mapper_update(const StationInputState &inputs) {
    StatusMapperState nextState;
    status_mapper_get_snapshot(nextState);

    for (uint8_t i = 0; i < MAX_PORT_COUNT; i++) {
        PortInputMapping &port = nextState.ports[i];
        port.enabled = (i < nextState.activePortCount);

        if (!port.enabled) {
            nextState.statuses[i] = PORT_STATUS_DISABLED;
            continue;
        }

        const bool errorActive = inputs.active[port.errorInput];
        const bool chargingActive = inputs.active[port.chargingInput];

        if (errorActive) {
            nextState.statuses[i] = PORT_STATUS_ERROR;
        } else if (chargingActive) {
            nextState.statuses[i] = PORT_STATUS_CHARGING;
        } else {
            nextState.statuses[i] = PORT_STATUS_WAITING;
        }
    }

    portENTER_CRITICAL(&mapperStateMux);
    mapperState = nextState;
    portEXIT_CRITICAL(&mapperStateMux);
}
```

### 10.2. Кто и когда вызывает функцию

Она вызывается:

- из `updatePortStatusFromInputs()` после каждого обычного чтения;
- из `startup_error_active()` во время стартовой анимации.

### 10.3. Какие данные существуют перед входом

- `inputs` содержит согласованный снимок восьми входов;
- `mapperState` содержит соответствие входов портам и предыдущие статусы;
- `activePortCount` равен 2;
- массивы имеют согласованные размеры.

Параметр объявлен так:

```cpp
const StationInputState &inputs
```

`&` передает структуру по ссылке без дополнительной полной копии. `const` запрещает функции менять входной снимок.

### 10.4. Построчное объяснение

#### Временное состояние

```cpp
StatusMapperState nextState;
```

Создается локальная структура. Она не обнуляется, но следующая строка сразу полностью копирует в нее опубликованный `mapperState`.

```cpp
status_mapper_get_snapshot(nextState);
```

Копируются `activePortCount`, соответствие входов всех портов и предыдущие статусы.

#### Цикл

```cpp
for (uint8_t i = 0; i < MAX_PORT_COUNT; i++) {
```

Обрабатываются все четыре возможных порта, включая зарезервированные.

#### Ссылка на соответствие входов одного порта

```cpp
PortInputMapping &port = nextState.ports[i];
```

Создается ссылка `port` на конкретный элемент массива.

Символ `&` здесь означает: `port` является другим именем для `nextState.ports[i]`. Изменение `port.enabled` изменяет поле внутри `nextState`.

Это не новый отдельный `PortInputMapping`.

#### Повторная проверка enabled

```cpp
port.enabled = (i < nextState.activePortCount);
```

Признак активности приводится в соответствие с `activePortCount`.

#### Зарезервированный порт

```cpp
if (!port.enabled) {
    nextState.statuses[i] = PORT_STATUS_DISABLED;
    continue;
}
```

Если порт не активен, его статус становится `disabled`.

Оператор `continue` немедленно переходит к следующей итерации цикла. Входы зарядки и ошибки зарезервированного порта дальше не читаются для вычисления статуса.

#### Чтение признака ошибки

```cpp
const bool errorActive = inputs.active[port.errorInput];
```

Берется логическая активность входа ошибки данного порта.

Для Port1 `port.errorInput = 1`, поэтому читается `inputs.active[1]`, то есть Data2.

#### Чтение признака зарядки

```cpp
const bool chargingActive = inputs.active[port.chargingInput];
```

Для Port1 `port.chargingInput = 0`, поэтому читается Data1.

#### Приоритет ошибки

```cpp
if (errorActive) {
    nextState.statuses[i] = PORT_STATUS_ERROR;
```

Ошибка проверяется первой. Если она активна, состояние сразу становится error.

```cpp
} else if (chargingActive) {
    nextState.statuses[i] = PORT_STATUS_CHARGING;
```

Зарядка проверяется только при отсутствии ошибки.

```cpp
} else {
    nextState.statuses[i] = PORT_STATUS_WAITING;
}
```

Если не активны ни ошибка, ни зарядка, порт ожидает.

Итоговый приоритет:

```text
error > charging > waiting
```

#### Публикация

```cpp
portENTER_CRITICAL(&mapperStateMux);
mapperState = nextState;
portEXIT_CRITICAL(&mapperStateMux);
```

Все четыре статуса и соответствия входов публикуются одной защищенной копией.

### 10.5. Какие переменные создаются и меняются

Локально:

- `nextState`;
- счетчик `i`;
- ссылка `port` на текущее соответствие входов;
- неизменяемые `errorActive` и `chargingActive` для активного порта.

Глобально в конце меняется `mapperState`.

Входной `inputs` не меняется благодаря `const`.

### 10.6. Что функция возвращает или публикует

Прямого возвращаемого значения нет. Она публикует новый `mapperState`.

### 10.7. Кто использует результат дальше

- `led_refresh_internal()` выбирает состояние зоны Port1..Port4;
- стартовая анимация ищет error;
- `/diagnostics` показывает статусы;
- `/get_config` получает число активных портов;
- `/save_zones` проверяет обязательные зоны активных портов.

### 10.8. Что произойдет при ошибке

Явного кода ошибки нет.

Безопасность индексов основана на текущем соответствии входов:

```text
4 порта * 2 входа = 8 входов
```

Если соответствие входов или размеры массивов будут изменены несогласованно, возможно чтение за границей массива. При текущих константах индексы находятся в диапазоне 0..7.

Электрический шум не считается программной ошибкой и может менять статус. Необходимость фильтрации определяется аппаратной проверкой.

## 11. `status_mapper_get_snapshot()`

Файл: `src/status_mapper.cpp`, текущие строки 55-59.

### 11.1. Полный актуальный код

```cpp
void status_mapper_get_snapshot(StatusMapperState &snapshot) {
    portENTER_CRITICAL(&mapperStateMux);
    snapshot = mapperState;
    portEXIT_CRITICAL(&mapperStateMux);
}
```

### 11.2. Кто и когда вызывает функцию

Ее вызывают:

- `status_mapper_update()` перед вычислением нового состояния;
- `startup_error_active()` для проверки ошибок;
- `led_refresh_internal()` для построения каждого кадра;
- `appendZoneMetadataJson()` для active/reserved зон;
- `zoneMapHasRequiredActivePortZones()` для проверки сохранения;
- `handleGetConfig()` для `activePortCount`;
- `handleDiagnostics()` для статусов портов.

### 11.3. Какие данные существуют перед входом

- `mapperState` содержит последнюю опубликованную структуру;
- вызывающая функция передает существующую переменную `StatusMapperState` по ссылке.

### 11.4. Построчное объяснение

```cpp
void status_mapper_get_snapshot(StatusMapperState &snapshot) {
```

Параметр `snapshot` является ссылкой на объект вызывающего кода. Символ `&` позволяет функции заполнить этот объект.

```cpp
portENTER_CRITICAL(&mapperStateMux);
```

Получается блокировка, чтобы `status_mapper_update()` не заменил структуру во время копирования.

```cpp
snapshot = mapperState;
```

Копируются число активных портов, четыре структуры соответствия входов и четыре статуса.

```cpp
portEXIT_CRITICAL(&mapperStateMux);
```

Блокировка сразу снимается.

### 11.5. Какие переменные создаются и меняются

Новых локальных переменных нет.

Меняется переданный `snapshot`. Глобальный `mapperState` только читается.

### 11.6. Что функция возвращает или публикует

Результат возвращается не оператором `return`, а записью в параметр по ссылке.

### 11.7. Кто использует результат дальше

Каждый вызывающий код работает со своей копией. Например, светодиодная задача может перебрать все 96 пикселей, не удерживая `mapperStateMux` все это время.

Это важная причина снимка: общая блокировка занята только на момент копирования, а не на весь расчет кадра или JSON.

### 11.8. Что произойдет при ошибке

Функция не возвращает ошибку. При корректном объекте-получателе копирование всегда выполняется.

Если состояние еще не было инициализировано, копия содержала бы начальные нули. В реальном порядке `setup()` сначала вызывает `status_mapper_setup()`, а уже затем запускает постоянных потребителей.

## 12. Временная шкала 0, 50, 100 и 150 мс

Ниже время отсчитывается условно от первого входа задачи в `inputTaskWorker()`. Реальный старт может произойти на несколько миллисекунд позже вызова `startInputTask()`.

Пусть:

```text
INPUT_TASK_PERIOD_MS = 50
lastWake при старте соответствует времени 0 мс
period = pdMS_TO_TICKS(50 мс)
```

### Нормальный короткий цикл

| Время | Что происходит | Значение `lastWake` после задержки |
|---:|---|---:|
| 0 мс | читаются Data1..Data8, вычисляются статусы | следующая цель 50 мс |
| около 1 мс | работа закончена, задача ждет | 50 мс |
| 50 мс | задача просыпается и снова читает входы | следующая цель 100 мс |
| 100 мс | третье чтение | следующая цель 150 мс |
| 150 мс | четвертое чтение | следующая цель 200 мс |

`pdMS_TO_TICKS()` нужен, потому что `vTaskDelayUntil()` работает не с миллисекундами, а с тиками планировщика (`tick`). Макрос переводит 50 мс в правильное число тиков для текущей конфигурации.

`lastWake` хранит не «сколько еще ждать», а опорный плановый момент предыдущего пробуждения. `vTaskDelayUntil()` прибавляет к нему `period` и ждет до полученного абсолютного момента.

### Отличие от обычной задержки

Если полезная работа каждый раз занимает 3 мс:

```text
обычная delay(50):
0 -> работа до 3 -> ожидание до 53 -> работа до 56 -> ожидание до 106

vTaskDelayUntil():
0 -> работа до 3 -> ожидание до 50 -> работа до 53 -> ожидание до 100
```

Второй вариант меньше накапливает сдвиг расписания.

### Если один цикл занял больше 50 мс

Предположим, первый вызов закончился только в 70 мс:

1. Плановая точка 50 мс уже прошла.
2. `vTaskDelayUntil()` не ждет еще полные 50 мс.
3. Следующая итерация может начаться сразу.
4. `lastWake` продолжает двигаться по сетке 0, 50, 100, 150 мс.
5. Если следующая работа короткая и завершится до 100 мс, задача подождет до 100 мс.

Если работа постоянно занимает больше периода, задача будет почти не спать. Новые вызовы не накладываются друг на друга, но фактический период чтения станет равен длительности работы. Это может отнять процессорное время у менее приоритетных задач.

В текущем пути используются короткие `digitalRead()`, копирование структур и несколько условий, поэтому нормальная длительность должна быть намного меньше 50 мс. Это утверждение окончательно подтверждается измерением на реальной плате.

## 13. Полный пример: Data1 = LOW, Data2 = HIGH, два активных порта

Исходные условия:

```text
DEFAULT_ACTIVE_PORT_COUNT = 2
BOARD_INPUT_ACTIVE_LOW = true

Data1 = LOW
Data2 = HIGH
```

Для полноты примера предположим, что остальные входы сейчас `HIGH`:

```text
Data3 = HIGH
Data4 = HIGH
Data5 = HIGH
Data6 = HIGH
Data7 = HIGH
Data8 = HIGH
```

### 13.1. Значения в `station_inputs_update()`

До цикла:

```text
nextState.rawLevel = [0, 0, 0, 0, 0, 0, 0, 0]
nextState.active   = [false, false, false, false, false, false, false, false]
```

Важно: числовое представление `LOW` обычно равно 0, а `HIGH` равно 1, но смысл в коде задают имена `LOW` и `HIGH`.

После `i = 0`, то есть Data1:

```text
level  = LOW
active = (LOW == LOW) = true

nextState.rawLevel[0] = LOW
nextState.active[0]   = true
```

После `i = 1`, то есть Data2:

```text
level  = HIGH
active = (HIGH == LOW) = false

nextState.rawLevel[1] = HIGH
nextState.active[1]   = false
```

После полного цикла:

```text
nextState.rawLevel[] = [LOW, HIGH, HIGH, HIGH, HIGH, HIGH, HIGH, HIGH]

nextState.active[] = [
    true,   // Data1 charging Port1 активен
    false,  // Data2 error Port1 не активен
    false,  // Data3 charging Port2
    false,  // Data4 error Port2
    false,  // Data5 charging Port3
    false,  // Data6 error Port3
    false,  // Data7 charging Port4
    false   // Data8 error Port4
]
```

После защищенного присваивания `inputState` содержит те же значения.

### 13.2. Содержимое `PortInputMapping`

После `status_mapper_setup()`:

```text
ports[0] - Port1
    chargingInput = 0
    errorInput    = 1
    zoneId        = 1
    enabled       = true

ports[1] - Port2
    chargingInput = 2
    errorInput    = 3
    zoneId        = 2
    enabled       = true

ports[2] - Port3
    chargingInput = 4
    errorInput    = 5
    zoneId        = 3
    enabled       = false

ports[3] - Port4
    chargingInput = 6
    errorInput    = 7
    zoneId        = 4
    enabled       = false
```

Начальные статусы после одного `status_mapper_setup()`:

```text
statuses[] = [
    PORT_STATUS_WAITING,
    PORT_STATUS_WAITING,
    PORT_STATUS_DISABLED,
    PORT_STATUS_DISABLED
]
```

### 13.3. Вычисление Port1

При `i = 0`:

```text
port = nextState.ports[0]
port.enabled = (0 < 2) = true

errorActive = inputs.active[port.errorInput]
            = inputs.active[1]
            = false

chargingActive = inputs.active[port.chargingInput]
               = inputs.active[0]
               = true
```

Проверка идет в порядке:

```text
if errorActive       -> false
else if chargingActive -> true
```

Результат:

```text
nextState.statuses[0] = PORT_STATUS_CHARGING
```

### 13.4. Вычисление Port2

При `i = 1`:

```text
port.enabled = (1 < 2) = true
errorActive = inputs.active[3] = false
chargingActive = inputs.active[2] = false
```

Результат:

```text
nextState.statuses[1] = PORT_STATUS_WAITING
```

### 13.5. Port3 и Port4

Для `i = 2` и `i = 3` условие `i < 2` ложно. Выполняется `continue`, поэтому их физические входы не влияют на опубликованный статус.

### 13.6. Итоговый `StatusMapperState`

```text
activePortCount = 2

statuses[] = [
    PORT_STATUS_CHARGING,  // Port1
    PORT_STATUS_WAITING,   // Port2
    PORT_STATUS_DISABLED,  // Port3
    PORT_STATUS_DISABLED   // Port4
]
```

Итоговый статус Port1: `PORT_STATUS_CHARGING`.

Светодиодная функция `led_refresh_internal()` получает снимок этого массива. Для каждого физического пикселя зоны Port1 она выбирает слой зарядки или запасной цвет `cfg.color_charge`, применяет общую яркость портов и передает цвет FastLED.

## 14. Почему ошибка всегда имеет приоритет

Для Port1 возможны четыре комбинации:

| Data1 charging | Data2 error | `chargingActive` | `errorActive` | Статус |
|---|---|---:|---:|---|
| HIGH | HIGH | false | false | waiting |
| LOW | HIGH | true | false | charging |
| HIGH | LOW | false | true | error |
| LOW | LOW | true | true | error |

Последняя строка важна. Даже если одновременно активен charging, первая ветка `if (errorActive)` уже выбрала error. Ветка charging не выполняется.

Такой приоритет находится в одном месте, в `status_mapper_update()`. Веб-интерфейс и свободные зоны не вычисляют `PortStatus` и не могут поменять этот порядок.

## 15. Где заканчивается ответственность каждой функции

| Функция | Ее ответственность | Чего она не делает |
|---|---|---|
| `station_inputs_setup()` | настроить GPIO и сделать первое чтение | не вычисляет статус порта |
| `station_inputs_update()` | прочитать восемь уровней и опубликовать `inputState` | не знает про Port1 и Port2 |
| `station_inputs_get_snapshot()` | безопасно скопировать последнее чтение | не читает GPIO заново |
| `status_mapper_setup()` | создать соответствие входов и модель активных/зарезервированных портов | не читает электрические уровни |
| `status_mapper_update()` | вычислить четыре `PortStatus` | не выбирает цвета светодиодов |
| `status_mapper_get_snapshot()` | безопасно скопировать статусы | не пересчитывает их |
| `updatePortStatusFromInputs()` | соединить чтение и mapper в одну операцию верхнего уровня | не задает период |
| `inputTaskWorker()` | повторять операцию по расписанию | не создает себя сама |
| `startInputTask()` | создать задачу и включить запасной признак | не читает входы непосредственно |

Это разделение помогает проверять систему по частям:

1. Диагностика `rawLevel` подтверждает электрический GPIO.
2. Диагностика `active` подтверждает принцип «активен низкий уровень» (active-low).
3. Диагностика `statuses` подтверждает соответствие входов и приоритет.
4. Физическая матрица подтверждает зоны, цвета и вывод FastLED.

## 16. Что требует проверки на реальной плате

По коду подтверждено:

- чтение выполняется примерно каждые 50 мс;
- Data1/Data2 относятся к Port1;
- Data3/Data4 относятся к Port2;
- Port1 и Port2 активны при `DEFAULT_ACTIVE_PORT_COUNT = 2`;
- ошибка проверяется раньше зарядки;
- чтение и публикация защищены короткими критическими секциями;
- при ошибке запуска задачи используется запасное чтение из `loop()`.

На железе нужно проверить:

- действительно ли активная оптопара дает устойчивый `LOW`;
- не плавают ли входы при отключенном кабеле;
- нет ли коротких импульсов и дребезга;
- достаточно ли периода 50 мс для реальных сигналов;
- не нужна ли фильтрация нескольких последовательных чтений;
- хватает ли стека 2048 байт при длительной работе;
- продолжается ли своевременная error-индикация во время Wi-Fi scan, OTA-запросов и диагностики;
- соответствуют ли GPIO фактической разводке платы.

Фильтрацию нельзя добавлять вслепую: слишком длинная задержка подтверждения может замедлить отображение ошибки. Сначала нужно измерить реальные сигналы.
